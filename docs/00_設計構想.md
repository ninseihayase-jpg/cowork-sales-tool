## 0. このツールが解く課題（背景）

現状、営業の状況管理は秘書プロジェクト（Hisho/Noct）の「テーマDB」で行っている。

- テーマDB = `todos` テーブルの `node_type='theme'` 行。**タスク管理の延長**として営業状況も乗っている。
- 入力は **主が手元の `sales_themes_input.xlsx` を編集 → `import_sales_xlsx.py` を手実行 → Render上のDBへ反映** という一方向・属人的フロー。
- 可視化は `dashboard.html`（Renderの `/dashboard`）で、判定KPI・パイプライン・コスト削減モデル等を高度に集計。

**課題**
1. xlsxを開いて編集 → スクリプト実行、という手間。**主しか編集できない**（属人化）。
2. テーマDBは「タスク管理」と「営業管理」が混在。本来、アカウント／商談の管理は独立した正本を持つべき。
3. 営業メンバー全員が、ストレスなく営業情報を入力・更新できる状態がない。

**目指す状態**
- アカウント管理・商談管理の**独立した営業情報DB**を持つ。
- そこへの入力が**スプレッドシート → ブラウザ画面 → Slack** と段階的に楽になる。
- 営業情報が**テーマDB／ダッシュボードへタイムリーかつ正しく自動連携**される。
- 最終的に営業メンバーは「ツールを開く・入力する」ストレスから解放される。

## 1. SFA/CRMの要点とリサーチ知見（設計の土台）

### 1.1 SFA/CRMの基本データモデル（業界標準）
営業支援ツールは概ね以下の4オブジェクトで構成される（Salesforce等の標準構造）。

| オブジェクト | 意味 | 本ツールでの呼称 |
|---|---|---|
| Account（取引先） | 顧客企業そのもの | **アカウント（企業）** |
| Contact（取引先責任者） | 企業の担当者個人 | **コンタクト（担当者）** |
| Opportunity（商談） | 1企業に複数立ちうる「案件」。ステージ・確度・金額・受注予定日を持つ | **商談（ディール）** |
| Activity（活動） | 面談・電話・メール等の履歴 | **活動（アクティビティ）** |

- 1アカウントに複数コンタクト、複数商談がぶら下がる（1対多）。
- 商談はステージ（リード→受注/失注）で進捗管理。これは**現テーマDBの `deal_stage` とほぼ同義**。

### 1.2 運用上の落とし穴（最重要・リサーチで裏付け済み）
SFA導入が失敗する最大要因は **「入力負荷 → 入力されない → データがバラつく → 使えないシステム化」** の悪循環。

1. **入力項目が多すぎる** → 記録に時間がかかり後回しに。
2. **現場にメリットがない** → 経営層は可視化したいが、現場は「入力が増えるだけ」と感じる。
3. **運用ルールの欠如** → 担当ごとに粒度・タイミング・記法がバラバラ。
4. **Excel/個人メモへの逃避** → SFAに転記されない。

**→ 本ツールの設計原則（全フェーズ共通）**
- **入力項目は最小限**。営業活動に直結する項目だけ。
- **選択肢はプルダウン/ボタン化**（手入力を減らし表記揺れを防ぐ）。テーマDBの選択肢セット（§3）を流用。
- **現場メリットを先に返す**：入力したらダッシュボードが更新され、滞留アラートや次アクションが返ってくる。
- 段階的に**入力チャネルを「営業マンが既にいる場所（Slack）」へ寄せる**（フェーズ2-2）。これはSalesforce自身もSlack統合（conversational CRM）で進める業界の本流。

### 1.3 あると効く機能（優先実装の候補）
- パイプライン可視化（ステージ別件数・金額）… **既存ダッシュボードが充足**。
- **滞留アラート**（一定日数ステージが動かない商談を通知）。
- 失注兆候フラグ、次アクション（次回マイルストン）管理 … 既存KPIに一部あり。

## 2. 現状システムの把握（連携先の正確な理解）

| 要素 | 実体 | 補足 |
|---|---|---|
| 正本DB | Render上の SQLite `/data/hisho.db` の `todos` テーブル | `node_type='theme'` がテーマDB。現在 **58テーマ**（Sales open 31 / Delivery open 9 等） |
| 入力 | `sales_themes_input.xlsx` → `scripts/import_sales_xlsx.py` | `/api/execute` 経由でDBへ INSERT/UPDATE（主が手実行） |
| 可視化 | `dashboard.html` → Render `/dashboard` | `/api/themes`(JSON) からデータ取得 |
| DB→スプシ | `/api/themes.csv?token=…` | Googleスプレッドシートの `IMPORTDATA` 用（読み取り専用で既に提供） |
| 配信層 | `src/hisho/debug_api.py`（標準ライブラリのHTTPサーバ） | `/dashboard` `/api/themes` `/api/themes.csv` `/api/execute` 等。token認証（実値は.env/Secrets管理） |
| デプロイ | Render `hisho-ohxe.onrender.com`（web, Singapore, autoDeploy） | GitHub `ninseihayase-jpg/Hayase-Secretary` と連携 |
| スケジューラ | `src/hisho/scheduler.py`（APScheduler） | 朝ブリーフィング等の定時ジョブ基盤。**ここに同期ジョブを足せる** |

**重要な連携の事実**
- すでに **DB → スプレッドシート方向（読み取り）** は `/api/themes.csv` で存在する。
- 今回フェーズ1で必要なのは **逆方向＝スプレッドシート（入力） → DB** の自動反映。
- DB書き込みは `/api/execute`（INSERT/UPDATE/DELETE/SELECT のみ許可、DDL不可、token認証）で外部から実行可能。**フェーズ1の同期はこのAPIを使えば、Render側のコード変更を最小化できる**。

## 3. テーマDBの選択肢セット（連携時のマスタ／表記揺れ防止）

営業情報DB→テーマDBへ連携する際、値はテーマDBの選択肢に揃える（`docs/db_schema_design.md` が正本）。主要なもの：

- **案件Stage（deal_stage）**: リード / アポ獲得 / 初回アポ実施 / 要件詰め / 提案 / クロージング / 受注 / 失注 / 保留中
- **分類（category）**: Sales / Delivery / プロダクト開発 / マーケ / 経営・戦略 / 長期・学習 / その他
- **事業種別L1**: コスト削減 / コンサルティング / AI導入 / 他
- **リード経路**: Connection / Exh. / Partner / Advisor / PE / Under / SNS / HP / na
- **企業規模**: 500億未満 / 1000億未満 / 5000億未満 / 5000億以上
- 金額の単位ルール：基本「万円」。**アプローチ額のみ「億円」**（L1=コスト削減専用）。

> 詳細・コスト削減モデルは秘書側 `docs/db_schema_design.md` を参照。営業情報DBはそのサブセット＋独自の活動履歴を持つ。

## 4. 全体アーキテクチャ構想

```
                 ┌─────────────────────────────────────────────────────┐
   入力チャネル  │  ① スプレッドシート(Google Sheets)  ← フェーズ1       │
   （段階的に     │  ② ブラウザ入力画面                 ← フェーズ2-1     │
    楽になる）    │  ③ Slack 商談メモ + AI            ← フェーズ2-2     │
                 └───────────────┬─────────────────────────────────────┘
                                 │ （同期：定時 or 即時）
                                 ▼
                 ┌─────────────────────────────────────────────────────┐
                 │   営業情報DB（独立）                                  │
                 │   accounts / contacts / deals / activities          │
                 └───────────────┬─────────────────────────────────────┘
                                 │ 連携（Sales分のみ集約・マッピング）
                                 ▼
                 ┌─────────────────────────────────────────────────────┐
                 │   テーマDB（既存 todos/theme, Render hisho.db）       │
                 │   ＝ タスク管理＋営業可視化。主が非Sales分を独自管理    │
                 └───────────────┬─────────────────────────────────────┘
                                 ▼
                 │   ダッシュボード（既存 /dashboard）自動反映           │
```

**設計の肝**
- 営業情報DB（アカウント/商談）は**独立した正本**。テーマDBとは別物として持つ。
- ただし**Salesの商談はテーマDBの「Salesテーマ」と1対1で対応**させ、商談更新→テーマDB同期→ダッシュボード反映を自動化する。
- 主が個人管理する**非Sales（経営・戦略・学習等）はテーマDB側でのみ管理**（営業情報DBには載らない）＝「スプシがSales/非Salesに分かれ、Salesは皆が、非Salesは主が編集」という運用イメージを、シート分割＋連携範囲の限定で実現する。

## 5. フェーズ計画

### フェーズ1 — スプレッドシート手編集 → DB自動反映（SFAツールなし）
**狙い**：xlsx属人運用を脱し、**全員がGoogle Sheetsを編集 → 自動でDB・ダッシュボードに反映**される状態。

- Google Sheets を入力の正本にする。シートを **「Sales」と「Sales以外」に分割**（Salesは皆が編集、Sales以外は主のみ編集という運用イメージ）。
- 列構成は現 `sales_themes_input.xlsx` の「テーマKPI」シートを踏襲（プレフィックスで入力主体を表現）。
- 同期方式（推奨）：**Render側のスケジューラ（APScheduler）に定時ジョブ**を追加し、Google Sheets API でシートを読み取り → 既存の取込ロジックでDBへ UPSERT。手動トリガ（Slackコマンド or APIエンドポイント）も用意。
  - 代替案：GitHub Actions の cron で同期（Renderに触らない）。または当面は「ローカルで `sync_sheet_to_db.py` を実行」でも可。
- 双方向衝突の回避：**入力シート（Sales/Sales以外）と、DB→スプシ表示シート（IMPORTDATA）を物理的に分ける**。入力シートが唯一の編集対象、表示シートは閲覧専用。
- **必要なユーザ作業**：Google Sheets作成、Google接続（サービスアカウント or OAuth）の許可。→ §8。

**完了の定義**：Sheetsを編集 → （定時 or トリガ）→ ダッシュボードに反映される、を実機で確認。

### フェーズ2 — 営業支援ツールで入力 → スプシ/DB自動反映

#### フェーズ2-1 — ブラウザ入力画面 ＋ 独立した営業情報DB
**狙い**：スプシ直編集をやめ、**ブラウザのフォームでアカウント/商談/活動を入力**。営業情報DBが正本になり、スプシ・テーマDBへ自動連携。

- 独立した営業情報DB（§6のスキーマ）。
- ブラウザ入力画面（アカウント一覧・商談カード・活動ログ・ステージ更新）。入力負荷最小・プルダウン中心。
- 連携：商談保存 → テーマDB（Salesテーマ）へ同期 → ダッシュボード反映。スプシは表示用にエクスポート。
- 挙動安定を重視し、まずは**手動保存・明示同期**から。リアルタイム性は後で詰める。

#### フェーズ2-2 — Slack商談メモ → AIクローリング → DB転記 ＋ 不足ヒアリング
**狙い**：営業メンバーは商談しながらSlackの専用チャンネルにメモを書くだけ。AIが定期的に読み取り、営業情報DBへ構造化転記し、**不足項目はSlackで質問**して埋める。

- 秘書プロジェクトの Slack基盤（Bolt/Socket Mode）＋Claude（tool use）を活用。
- フロー：チャンネル監視 → メモをLLMで構造化（企業/商談/ステージ/金額/次アクション抽出）→ 営業情報DBへ起票・更新 → 欠落項目をSlackで1問1答ヒアリング → 確定。
- **Go/NoGo判断ポイント**：Slack APIの制約（チャンネル履歴取得、メッセージ量、レート制限、誤認識時の訂正フロー、誰の発言を誰の商談に紐付けるか）。フェーズ2-1まで安定後に小さく検証してから本実装。

## 7. テーマDB連携設計（営業情報DB → テーマDB）

- **対応キー**：`deals.theme_id` ↔ `todos.id`。新規商談はテーマDBに対応テーマを作成し、採番されたidを `theme_id` に保存。
- **同期方向**：原則 **営業情報DB → テーマDB（Sales分）** の片方向。テーマDBの非Sales行は触らない。
- **同期項目**：deal_name / stage→deal_stage / business_type / lead_pattern / owner→deal_owner / 金額各種 / next_milestone→milestone / activities→meeting_dates / note / goal / industry / company_size。
- **同期手段**：既存 `/api/execute`（UPDATE/INSERT）を流用。`import_sales_xlsx.py` のマッピングを共通化して再利用。
- **冪等性**：同期は theme_id をキーに UPSERT。多重実行しても重複しない設計。
- **衝突方針**：Salesテーマは営業情報DBを正本とする（テーマDB側で直接編集しない運用）。主が非Sales＋テーマ独自項目（重要度・タスク分解）はテーマDB側で管理。

## 8. ユーザ（主）が対応すべき接続・権限（序盤の洗い出し）

| # | 項目 | 必要フェーズ | 主の対応 | Claude側でできること |
