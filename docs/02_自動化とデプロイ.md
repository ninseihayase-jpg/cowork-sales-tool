# 自動化とデプロイ

> フェーズ1の「定時的に読み解いて自動更新」を実現する選択肢と、独立Renderサービスの構成。

---

## 1. 同期の自動実行（フェーズ1）

スプレッドシート → テーマDB の同期 `scripts/sync_cli.py` を定期実行する方法。挙動安定を最優先に、まずは **A or B** を推奨。

### A. GitHub Actions の cron（推奨・最も手軽）
- このリポをGitHubに置き、Actionsで定時実行（例：30分ごと／1日数回）。
- 鍵JSON・トークンは GitHub の **Secrets** に登録（リポには置かない）。
- メリット：Renderに触らない／ログが残る／無料枠で足りる。
- ワークフロー雛形は `.github/workflows/sync.yml`（後述・作成済み）。

### B. ローカル/手動実行（最初の運用）
- 主または担当が `python scripts/sync_cli.py` を必要時に実行。
- まずはこれで運用を回し、頻度の感覚を掴んでからAへ自動化。

### C. 独立Renderサービスで常駐スケジューラ（将来）
- 本ツールが独立Renderサービスを持つ段階（フェーズ2-1で入力画面を載せるとき）に、
  APScheduler等で定時同期ジョブを内包。秘書プロジェクトの `scheduler.py` と同型。

---

## 2. GitHub Actions ワークフロー（雛形）

`.github/workflows/sync.yml` を同梱。必要なSecrets：
- `THEME_API_TOKEN` … テーマDB APIトークン（秘書プロジェクトのdebug API token。実値はSecretsに登録）
- `SALES_SHEET_ID` … スプレッドシートID
- `GOOGLE_SERVICE_ACCOUNT_JSON` … 鍵JSONの中身まるごと（複数行可）

> cronの `schedule` はUTC基準。日本時間で動かしたい時刻からは-9時間して指定する。

---

## 3. 独立Renderサービス（フェーズ2-1以降）

- 営業支援ツールは秘書(hisho)とは**別のRenderサービス／別GitHubリポ**として持つ（方針決定済み）。
- テーマDBへは秘書の `/api/execute` `/api/themes.csv` 経由で疎結合連携。
- フェーズ2-1でブラウザ入力画面を載せる際に web サービスとして `render.yaml` を用意する。
  - 営業情報DB（accounts/contacts/deals/activities）は独自のSQLite（Render Disk）に持つ。
  - 入力画面・営業DB→テーマDB同期もこのサービス内に実装。

---

## 4. ローカル開発・コミットの流れ

```bash
# 依存（venvが作れない環境では --user）
pip install --user -r requirements.txt

# 動作確認（DBに書かない）
python scripts/sync_cli.py --xlsx <既存xlsxパス> --dry-run

# 変更をコミット
git add -A && git commit -m "..."
git push    # GitHub連携後
```
